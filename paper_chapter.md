# 第X章 三维重建任务控制台系统设计与实现

## X.1 引言

三维高斯泼溅（3D Gaussian Splatting, 3DGS）作为一种新兴的显式场景表示方法，在三维重建领域展现出显著优势。然而，现有的3DGS推理系统通常以命令行脚本形式运行，缺乏友好的用户交互界面与任务管理能力，导致非专业用户在参数配置、任务提交、进度监控与结果查看等环节面临较高使用门槛。为此，本章设计并实现了一套面向DepthSplat三维重建推理引擎的Web任务控制台系统（以下简称"任务控制台"），旨在通过浏览器端提供一站式的三维重建任务管理服务。

该系统采用前后端分离架构，本章重点论述前端任务控制台的设计与实现，涵盖系统架构、核心模块设计、关键技术方案及性能优化等方面。

## X.2 系统架构设计

### X.2.1 整体架构

任务控制台采用B/S（Browser/Server）架构，前端基于React框架构建单页应用（SPA），通过HTTP RESTful接口与后端DepthSplat推理服务进行数据交互。系统整体架构如图X.1所示，自上而下分为四层：表示层（View Layer）、业务逻辑层（Business Logic Layer）、服务层（Service Layer）和数据持久层（Persistence Layer）。

- **表示层**：负责用户界面渲染与交互，采用Ant Design组件库构建响应式布局，包含任务创建、任务详情、历史记录与系统设置四个核心页面。
- **业务逻辑层**：封装业务状态管理与数据流控制，通过React Query实现服务端状态同步，利用自定义Store实现客户端状态持久化。
- **服务层**：封装HTTP通信与API调用，提供类型安全的请求/响应转换，对后端返回数据进行归一化处理。
- **数据持久层**：利用浏览器localStorage实现用户配置与任务历史记录的客户端持久存储。

### X.2.3 目录结构设计

项目采用功能分层的目录组织方式，将代码按职责划分为页面（pages）、组件（components）、钩子（hooks）、服务（services）、状态管理（store）、类型定义（types）与工具函数（utils）七个模块，实现关注点分离与模块间低耦合。核心目录结构如下：

```
src/
├── pages/          # 页面级组件（路由入口）
├── components/     # 可复用UI组件
├── hooks/          # 自定义React Hooks
├── services/       # HTTP通信与API封装
├── store/          # 客户端状态管理
├── types/          # TypeScript类型定义
└── utils/          # 工具函数与常量
```

## X.3 核心模块设计

### X.3.1 任务提交模块

任务提交模块是系统的入口功能，负责收集用户配置参数并提交至后端推理服务。该模块由TaskCreatePage页面、SampleSelector组件、VideoUploader组件及相关Hooks协同构成。

**（1）预设与样本选择机制**

系统将推理模式抽象为"预设（Preset）"与"样本（Sample）"两级选择机制。预设定义了推理模型与脚本的组合（如large、base、small三种规模），样本则指向预置的输入数据集。用户既可通过SampleSelector从后端获取的样本列表中选择已有样本，也可通过VideoUploader自主上传视频文件作为输入。

**（2）表单数据组装与提交**

任务创建采用FormData格式提交，支持同时上传多张输入图像与单个视频文件。提交时，系统将预设ID、样本ID、用户自定义选项（如test_chunk_interval、save_video、compute_scores、export_depth_map等）一并封装为请求负载，通过POST请求发送至后端推理服务。

### X.3.2 实时任务监控模块

实时任务监控是系统的核心功能，实现了对推理任务全生命周期的状态跟踪与日志展示。

**（1）轮询式状态同步机制**

鉴于后端服务未提供WebSocket长连接支持，系统采用基于React Query的轮询（Polling）策略实现准实时状态更新。具体而言，任务详情接口（`GET /tasks/:id`）以3秒间隔刷新，日志接口（`GET /tasks/:id/logs`）以2秒间隔刷新。通过refetchInterval参数控制轮询频率，在任务状态稳定（success/failed/cancelled）后自动停止轮询，避免不必要的网络开销。

**（2）任务状态模型**

系统定义了完整的任务状态机，涵盖QUEUED（排队中）、PREPARING（准备中）、RUNNING（运行中）、POSTPROCESSING（后处理中）、SUCCESS（成功）、FAILED（失败）、CANCELLED（已取消）与MISSING（已丢失）八种状态。状态转移遵循严格约束：仅活动状态（QUEUED、PREPARING、RUNNING、POSTPROCESSING）可转移至终止状态（SUCCESS、FAILED、CANCELLED），终止状态不可逆。

**（3）智能日志展示**

LogViewer组件实现了针对推理日志的定制化展示优化：
- **进度行去重**：对连续的进度输出（如百分比进度条）进行合并，仅展示最新进度，减少视觉干扰。
- **ANSI转义码过滤**：自动剥离后端日志中可能携带的ANSI颜色控制字符，确保Web端文本正常渲染。
- **日志分级着色**：根据日志级别（DEBUG/INFO/WARNING/ERROR）应用差异化颜色，提升可读性。

**（4）阶段耗时分解**

系统解析任务详情中的阶段时间戳字段，计算并展示各处理阶段（data_load、data_prep、inference、splat_convert、render、postprocess、score、export）的精确耗时，为用户提供细粒度的性能分析视图。

### X.3.3 结果可视化模块

ResultViewer组件负责对推理产出的多模态结果进行结构化展示：

- **输入图像墙**：以网格布局展示所有输入源图像。
- **输出视频播放器**：集成HTML5 video元素，展示重建结果视频。
- **深度图画廊**：以缩略图网格展示各视角的深度估计结果。
- **指标数据面板**：以描述列表（Descriptions）形式展示PSNR、SSIM、LPIPS等重建质量评估指标。
- **Splat Viewer集成**：生成指向外部3D高斯泼溅查看器（基于antimatter15/splat项目）的链接，用户可实时查看三维重建的点云效果。

### X.3.4 任务历史管理模块

由于后端未提供任务列表查询接口，系统自主实现了基于localStorage的客户端任务历史记录方案。

**（1）历史记录存储策略**

采用容量上限策略（默认200条），当历史记录超过上限时自动移除最早记录以控制存储开销。每条历史记录包含任务ID、状态、预设信息、样本信息及创建时间戳。

**（2）历史状态自动同步**

系统定时（3秒间隔）对历史列表中状态处于"活动"（非终止状态）的任务向后端发起状态查询，同步更新其状态至最新值。该机制确保用户在重新打开页面后仍能获得任务的最新状态。

**（3）多维过滤查询**

提供按任务状态、预设类型及时间范围的组合过滤能力，过滤逻辑在客户端完成，无需后端查询接口支持。

### X.3.5 设置管理模块

SettingsForm组件提供后端服务地址、任务选项开关及UI偏好设置的可视化配置界面。所有设置通过settingsStore持久化至localStorage（存储键：`depthsplat-settings`），页面加载时自动恢复上次配置，保证用户体验的连续性。

## X.4 关键技术方案

### X.4.1 前后端数据归一化方案

DepthSplat后端接口在迭代过程中存在字段命名不一致（snake_case与camelCase混用）的问题。为提升前端代码的健壮性，系统在服务层（tasks.ts）实现了自适应字段归一化机制：对每个响应对象的关键字段，同时尝试读取两种命名风格的对应值，优先取有效值。该方案使得前端能够兼容不同版本的后端接口，无需因后端命名变更而频繁调整前端代码。

### X.4.2 会话连续性设计

为保持用户操作连贯性，系统设计了基于内存单例Store的当前任务会话机制（currentTaskStore）。当用户从任务创建页跳转至任务详情页时，当前任务ID、预设ID、样本ID等上下文信息通过Store保持而不丢失，避免了跨页面导航时重复查询或状态丢失的问题。

### X.4.3 请求防抖与组件卸载保护

在任务详情轮询场景中，系统利用React Query的enabled参数实现条件性数据获取：仅当路由中存在有效taskId参数时才启用查询，且对异常状态（如网络错误）进行静默捕获，避免在组件卸载后仍尝试更新状态引发内存泄漏。

## X.5 系统运行效果

任务控制台在本地开发环境（端口5174）经功能验证测试，各功能模块运行正常。图X.2展示了任务创建界面，用户可在此完成预设选择、样本配置与视频上传操作。图X.3展示了任务详情界面，包含状态卡片、实时日志窗口与结果展示区域。图X.4展示了历史记录页面，支持多维过滤与状态同步。

系统在实际使用中表现出良好的稳定性：单任务从提交到完成的全流程监控延迟可控（轮询间隔3秒），日志刷新频率（2秒）满足实时性需求，客户端历史存储上限（200条）满足日常使用场景。

## X.6 本章小结

本章设计并实现了一套面向DepthSplat三维重建推理引擎的Web任务控制台系统。系统采用React + TypeScript + Vite技术栈，基于Ant Design构建企业级用户界面，通过React Query实现轮询式实时监控，利用localStorage实现客户端数据持久化。系统覆盖了任务创建、实时监控、结果可视化和历史管理四大功能域，通过数据归一化、会话连续性等关键技术方案保障了系统的健壮性与用户体验。该系统的实现为三维重建推理引擎提供了直观、高效的Web操作入口，降低了非专业用户的使用门槛，具有较强的工程应用价值。
